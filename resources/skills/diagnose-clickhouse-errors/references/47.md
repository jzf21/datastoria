# Code 47: UNKNOWN_IDENTIFIER

The query references a column, alias, or expression name that is not visible in the current scope.

## Workflow

1. Extract the unknown identifier from the error text.
   - Prefer the name inside `Unknown expression identifier \`...\``
   - Or the first name after `Missing columns: ...`
2. Extract the query fragment after `in scope` when present.
3. Treat this mainly as a scope issue. Common causes:
   - typo in a column or alias name
   - wrong table alias in a `JOIN`
   - column not projected from a subquery or CTE
   - alias used in a clause where it is not visible
   - ungrouped column referenced after aggregation
4. If the query clearly references a qualified table as `database.table`, call `execute_sql` with the SQL below, replacing `DATABASE_NAME`, `TABLE_NAME`, and `IDENTIFIER_NAME` with single-quoted strings; otherwise skip to step 6. Do not modify the query or call `execute_sql` again.

    ```sql
    SELECT
        name,
        type,
        levenshteinDistance(lower(name), lower('IDENTIFIER_NAME')) AS dist
    FROM system.columns
    WHERE database = 'DATABASE_NAME' AND table = 'TABLE_NAME'
    ORDER BY dist
    LIMIT 3
    ```

5. Interpret the result:
   - `dist = 0`: the column exists in this table; the problem is a scope, alias, or projection issue — use the causes in step 3 to identify which.
   - `dist > 0`: likely a typo; list all returned matches with their `type` so the user can pick the right one.
   - No rows: the column may not exist; diagnose from the query text.
6. If no qualified table was found or `execute_sql` fails, diagnose from the query text using the causes in step 3 as a checklist.
7. Present: error `47` / `UNKNOWN_IDENTIFIER`, the unknown identifier, whether it is likely a typo or a scope issue, the most likely fix, and one fenced SQL example when the fix is clear.
